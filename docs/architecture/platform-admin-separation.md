# فصل لوحة إدارة سند إلى `admin.sanadflow.com`

## القرار

تعتمد سند مسار فصل تدريجي للوحة مدير المنصة إلى تطبيق واجهة مستقل على:

```text
https://admin.sanadflow.com
```

مع إبقاء قاعدة البيانات، Supabase Auth، وظائف الإدارة وEdge Functions مشتركة مع تطبيق سند الأساسي.

لا يتطلب هذا القرار خادم قاعدة بيانات جديدًا، ولا نسخة Supabase جديدة، ولا نقل منطق الإدارة إلى نظام منفصل.

## لماذا الفصل مناسب

- إزالة حزمة واجهة الإدارة الثقيلة من التطبيق العام للمستخدمين.
- منع اختلاط تجربة العميل أو صاحب النشاط بأدوات تشغيل المنصة.
- تشديد حماية النطاق الإداري والجلسات بصورة مستقلة.
- تمكين نشر لوحة الإدارة وتطويرها دون التأثير المباشر في PWA العام.
- توفير مساحة مناسبة للجداول الكبيرة، المراجعات، سجل التدقيق ومراقبة واتساب.
- إبقاء الوصول إلى الإدارة مشروطًا بصلاحية `platform admin` الفعلية، لا بمجرد معرفة الرابط.

## النموذج المعماري

```text
app.sanadflow.com
  ├─ تجربة المستخدم
  ├─ إدارة النشاط التجاري
  ├─ الملف العام والكتالوج
  └─ رابط دخول إداري يظهر للمدير فقط

admin.sanadflow.com
  ├─ تسجيل الدخول عبر Supabase Auth نفسه
  ├─ فحص getPlatformAdminAccess عند كل تحميل
  ├─ المستخدمون
  ├─ العمليات
  ├─ الأنشطة والمراجعات
  ├─ سند Pro والمدفوعات
  ├─ مستخدمو واتساب والحملات
  ├─ الإعدادات العامة
  └─ سجل الإدارة والتدقيق

Supabase
  ├─ Auth مشترك
  ├─ RPCs إدارية محمية
  ├─ RLS وحظر الوصول المباشر
  └─ audit log مشترك
```

## مراحل التنفيذ

### المرحلة 1 — جاهزية التطبيق الحالي

- إضافة `VITE_PLATFORM_ADMIN_URL`.
- اعتماد `https://admin.sanadflow.com` عنوانًا قانونيًا للوحة الإدارة.
- إبقاء المسار الداخلي `/platform-admin` مؤقتًا لأغراض الانتقال والتراجع.
- استمرار فحص الصلاحية داخل واجهة الإدارة نفسها.

### المرحلة 2 — إنشاء تطبيق Admin مستقل

ينشأ تطبيق Vite/React مستقل داخل المستودع أو في مستودع خاص، مع تفضيل monorepo في البداية:

```text
apps/
  app/
  admin/
packages/
  admin-api/
  auth/
  ui/
  types/
```

لا تنسخ دوال الإدارة يدويًا. تُنقل الأنواع وطبقة API المشتركة إلى package داخلي.

### المرحلة 3 — المصادقة والنطاق

- النطاق الإداري يستخدم Supabase Auth نفسه.
- إعداد redirect URLs للنطاق الإداري في Supabase.
- بعد استعادة الجلسة، ينفذ التطبيق `getPlatformAdminAccess()` قبل تحميل أي بيانات.
- عند رفض الوصول، تمسح بيانات الإدارة من الذاكرة وتظهر صفحة منع وصول.
- لا تستخدم قائمة بريد ثابتة أو شرطًا داخل الواجهة لمنح الصلاحية.

### المرحلة 4 — حماية وتشغيل

- Content Security Policy مستقلة.
- منع الفهرسة عبر `robots.txt` و`X-Robots-Tag: noindex`.
- تفعيل MFA/Passkey إلزامي لاحقًا لمديري المنصة.
- وقت انتهاء جلسة أقصر للإدارة.
- تسجيل كل mutation إداري مع السبب والفاعل والتوقيت.
- عدم حفظ بيانات حساسة في localStorage.

### المرحلة 5 — إزالة المسار القديم

بعد نجاح التطبيق الإداري في الإنتاج:

- بطاقة «لوحة إدارة سند» تفتح النطاق الإداري في تبويب جديد.
- `/platform-admin` يعيد التوجيه إلى `VITE_PLATFORM_ADMIN_URL`.
- إزالة lazy chunk الخاص بالإدارة من التطبيق العام.
- الإبقاء على fallback مؤقت لمدة إصدار واحد فقط.

## متغيرات البيئة

التطبيق العام:

```env
VITE_PUBLIC_APP_URL=https://app.sanadflow.com
VITE_PLATFORM_ADMIN_URL=https://admin.sanadflow.com
```

تطبيق الإدارة:

```env
VITE_PUBLIC_APP_URL=https://app.sanadflow.com
VITE_ADMIN_APP_URL=https://admin.sanadflow.com
VITE_SUPABASE_URL=...
VITE_SUPABASE_ANON_KEY=...
```

## حدود الأمان

إخفاء بطاقة الإدارة عن غير المدير هو تحسين تجربة فقط، وليس حاجز حماية. الحماية الملزمة تبقى داخل:

- RPCs الإدارية.
- `getPlatformAdminAccess()`.
- RLS ومنع القراءة المباشرة.
- سجل التدقيق.

أي تطبيق مستقل يجب أن يفترض أن المستخدم يستطيع كتابة عنوان الإدارة يدويًا، ثم يمنعه من البيانات والإجراءات على مستوى الخادم.

## قرار النشر

لا يُنشأ خادم تطبيق جديد في المرحلة الأولى. يمكن نشر Admin كتطبيق static مستقل عبر Nginx على الخادم الحالي:

```text
/var/www/admin.sanadflow.com/html
```

مع server block ونطاق وشهادة TLS مستقلين. هذا فصل نشر وواجهة، وليس فصل بيانات أو بنية خلفية.
