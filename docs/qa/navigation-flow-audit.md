# مراجعة سجل الحركة والتنقل بين الواجهات

## الهدف

تثبيت عقد تنقل واحد لتطبيق سند يمنع اختلاف حالة React عن عنوان المتصفح، ويضمن عمل الرجوع والتقدم والروابط المباشرة بصورة متوقعة.

## المشكلات المرصودة في البنية الحالية

- منطق تحليل المسار مكرر بين التحميل الأول وحدث `popstate`.
- بعض الواجهات تعدل `history` داخليًا، بينما واجهات أخرى تعتمد على `onNavigate`.
- بعض الانتقالات تستخدم `pushState`، وأخرى تستخدم `replaceState` دون قاعدة موحدة.
- بيانات `activeToken` و`activeProductToken` قد تبقى من صفحة سابقة إذا لم تنظف عند كل انتقال.
- تبويبات الملف العام تعدل query params داخليًا، بينما App يدير pathname.
- المسارات القديمة للكتالوج تحتاج تحويلًا قانونيًا واحدًا لا يتكرر في أكثر من موضع.
- الانتقال إلى واجهة عامة ثم الرجوع قد يعيد المستخدم إلى صفحة مختلفة إذا فُتحت الواجهة من رابط خارجي.

## عقد التنقل المقترح

كل وجهة يجب أن تمثل ككائن واحد:

```ts
type AppRoute =
  | { page: 'home' }
  | { page: 'profile'; section?: string }
  | { page: 'details'; token: string; source: 'link' | 'qr' | 'search' | 'app' }
  | { page: 'business-manage' }
  | { page: 'business-manage-profile' }
  | { page: 'public-business-profile'; slug: string; section?: string }
  | { page: 'public-product-detail'; slug: string; productId: string }
  | { page: 'platform-admin' };
```

ثم تعتمد ثلاث دوال فقط:

```text
parseLocation(location) → AppRoute
buildLocation(route) → pathname + search
applyRoute(route) → تحديث جميع حالات React المرتبطة
```

## قواعد push وreplace

استخدم `pushState` عندما:

- يفتح المستخدم صفحة أو قسمًا جديدًا بإجراء صريح.
- يتوقع أن يعيده زر الرجوع إلى الحالة السابقة.

استخدم `replaceState` عندما:

- نصحح مسارًا قديمًا.
- ننظف query parameter مؤقتًا.
- نزامن القسم الأولي دون إنشاء خطوة رجوع وهمية.

## قواعد تنظيف الحالة

عند كل انتقال يجب ضبط جميع القيم، لا القيمة المستخدمة فقط:

```text
activeToken
activeProductToken
activeSource
profileInitialTab
currentPage
```

أمثلة:

- العودة إلى Home تمسح token والمنتج والمصدر.
- فتح ملف نشاط يمسح product token.
- فتح منتج يضبط slug وproductId معًا.
- فتح الإدارة يمسح أي token عام سابق.

## سيناريوهات QA الإلزامية

### التطبيق العام

- [ ] Home → حسابي → رجوع.
- [ ] حسابي → إدارة النشاط → رجوع.
- [ ] إدارة النشاط → الملف العام → رجوع.
- [ ] الملف العام → عنصر منتج → رجوع إلى التبويب السابق.
- [ ] رابط مباشر `/b/{slug}` يعمل بعد تحديث الصفحة.
- [ ] رابط مباشر `/b/{slug}/p/{id}` يعمل بعد تحديث الصفحة.
- [ ] منتج غير موجود يعود بأمان إلى النشاط.
- [ ] المسار القديم للكتالوج يتحول مرة واحدة دون loop.

### تبويبات الملف العام

- [ ] تغيير التبويب يحدث query params.
- [ ] زر الرجوع يعيد التبويب السابق إذا استُخدم `pushState`.
- [ ] تحديث الصفحة يحتفظ بالقسم المحدد.
- [ ] المشاركة تنسخ رابط القسم الحالي.
- [ ] العودة إلى الغلاف لا تترك `section` قديمًا في الرابط.

### إدارة النشاط

- [ ] فتح قائمة الأقسام لا يضيف history entry.
- [ ] اختيار قسم يغلق النافذة ويحدث المحتوى.
- [ ] الرجوع من محرر الملف العام يعود إلى إدارة النشاط.
- [ ] عرض الملف العام يفتح رابطًا قانونيًا في تبويب مستقل.

### الطلب

- [ ] فتح وإغلاق Sheet لا يغير عنوان الصفحة.
- [ ] الرجوع في المتصفح أثناء فتح Sheet لا يترك scroll lock.
- [ ] الانتقال إلى منتج لا يفقد السلة المحلية.
- [ ] الرجوع إلى النشاط يعيد موضع التبويب بصورة مقبولة.

### الإدارة

- [ ] غير المدير لا يرى بطاقة الإدارة.
- [ ] كتابة `/platform-admin` يدويًا لا تمنح الوصول.
- [ ] عند تفعيل النطاق المستقل تفتح البطاقة `admin.sanadflow.com`.
- [ ] انتهاء الجلسة داخل الإدارة يعيد المستخدم إلى تسجيل الدخول.

## التحسين التنفيذي التالي

تُنقل دوال تحليل وبناء المسارات من `App.tsx` إلى:

```text
src/lib/appRoutes.ts
```

ثم يستبدل فرعا التحميل الأول و`popstate` باستدعاء واحد لـ`applyRoute(parseLocation())`.

لا يتم هذا التغيير بالتوازي مع تعديلات واجهات أخرى، لأنه يمس كل مسارات التطبيق ويحتاج بوابة اختبار مستقلة.
